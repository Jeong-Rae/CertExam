# SAA-C03 Practice App SPEC

## 1. 제품 개요

SAA-C03 Practice App은 AWS Certified Solutions Architect - Associate, 이하 SAA-C03 시험을 준비하는 사용자가 모바일 환경에서 문제를 풀고, 힌트를 확인하고, 정답·오답 해설을 통해 반복 학습할 수 있도록 돕는 문제풀이 앱이다.

이 앱은 단순한 퀴즈 앱이 아니라, 사용자가 실제 시험과 유사한 사고 흐름으로 문제를 읽고, 요구사항을 분석하고, 선택지를 비교하며, 오답을 복습하는 학습 루프를 제공하는 것을 목표로 한다.

상세 UI 스타일, 색상, 타이포그래피, 컴포넌트 디자인은 `design.md`에서 정의한다. 본 문서는 UX 흐름, 기능 요구사항, 사용자 Use Case, 도메인 모델, 정책, 상태 전이를 중심으로 설명한다.

---

## 2. 제품 목표

### 2.1 핵심 목표

사용자가 SAA-C03 문제를 빠르게 풀고, 틀린 문제를 구조적으로 복습하며, 시험 준비 과정에서 자신의 취약 영역을 점진적으로 줄일 수 있게 한다.

### 2.2 세부 목표

1. 사용자가 앱 진입 즉시 학습 현황을 파악하고 문제 풀이를 시작할 수 있어야 한다.
2. 문제 풀이 중 불필요한 시각적 방해 없이 문제 지문과 선택지에 집중할 수 있어야 한다.
3. 힌트는 정답을 직접 알려주는 것이 아니라, 사용자가 요구사항을 분석하도록 유도해야 한다.
4. 정답 여부는 즉시 피드백되어야 하며, 선택지별 해설을 통해 개념 이해로 이어져야 한다.
5. 오답은 자동으로 저장되어 이후 복습 흐름으로 자연스럽게 연결되어야 한다.
6. 사용자의 진행률, 정답률, 오답 수, 남은 문제 수를 기반으로 학습 상태를 명확히 보여줘야 한다.

---

## 3. 주요 사용자

### 3.1 Primary User: SAA-C03 시험 준비자

AWS SAA-C03 자격증을 준비하는 개인 학습자다.

주요 니즈는 다음과 같다.

* 출퇴근, 휴식 시간 등 짧은 시간에 문제를 풀고 싶다.
* 긴 지문에서 핵심 요구사항을 빠르게 파악하고 싶다.
* 왜 특정 선택지가 정답인지 명확히 이해하고 싶다.
* 반복적으로 틀리는 영역을 확인하고 복습하고 싶다.
* 전체 학습 진행 상황을 한눈에 보고 싶다.

### 3.2 Secondary User: 재학습 사용자

이미 일부 문제를 풀었고, 오답 또는 북마크한 문제를 중심으로 반복 학습하려는 사용자다.

주요 니즈는 다음과 같다.

* 이전에 틀린 문제만 다시 보고 싶다.
* 헷갈렸던 문제를 북마크해두고 나중에 확인하고 싶다.
* 단순 정답 암기가 아니라, 선택지별 판단 근거를 복습하고 싶다.

## 4. UX 원칙

### 4.1 문제 집중 우선

문제 풀이 화면에서는 문제 지문, 선택지, 진행 상태, 핵심 액션만 노출한다. 사용자가 문제를 읽고 판단하는 흐름에 방해되는 장식, 과도한 메뉴, 불필요한 통계 노출은 배제한다.

### 4.2 즉각적 피드백

답안을 제출하면 정답 또는 오답 상태를 즉시 보여준다. 결과 화면은 단순히 맞고 틀림을 알려주는 것에 그치지 않고, 왜 맞았는지 또는 왜 틀렸는지를 이해시키는 데 집중한다.

### 4.3 힌트는 정답 노출이 아니라 사고 유도

힌트는 사용자가 문제 요구사항을 정리하고, 부적절한 선택지를 제거하며, AWS 서비스의 특성을 떠올릴 수 있도록 돕는다. 정답 자체를 직접적으로 알려주지 않는다.

### 4.4 오답은 자동 저장

사용자가 틀린 문제는 별도의 수동 저장 없이 자동으로 오답 노트에 기록된다. 사용자는 학습 흐름을 끊지 않고 다음 문제로 이동할 수 있으며, 이후 오답 노트에서 누적된 약점을 복습할 수 있다.

### 4.5 학습 흐름 유지

정답 또는 오답 해설 확인 이후에는 명확한 다음 액션이 제공되어야 한다. 기본 액션은 `다음 문제`이며, 오답인 경우 `오답노트 보기`를 보조 액션으로 제공한다.

## 5. 주요 화면 및 상태

상세한 시각 디자인은 `design.md`를 따른다.

### 5.1 시작 화면

사용자가 앱을 실행했을 때 가장 먼저 보는 홈 화면이다.

주요 목적은 다음과 같다.

* 전체 문제 수 확인
* 현재 학습 진행률 확인
* 맞힌 문제, 틀린 문제, 남은 문제 확인
* 이어 풀기 또는 새 문제 풀이 시작
* 오답 노트, 힌트, 요약 노트, 통계 등 주요 기능 접근

핵심 기능:

* 학습 진행률 표시
* 최근 풀이 위치 기반 이어하기
* 전체 문제 수 표시
* 정답 수, 오답 수, 남은 문제 수 요약
* 문제 풀이 시작
* 오답 노트 진입
* 북마크 또는 요약 노트 진입

UX 기준:

* 사용자가 “지금 어디까지 했는지”와 “무엇을 하면 되는지”를 3초 안에 이해할 수 있어야 한다.
* 주요 CTA는 `문제 풀기`다.
* 오답 노트는 보조 CTA로 제공한다.

---

### 5.2 문제 풀이 화면

사용자가 문제를 읽고 선택지를 고르는 기본 풀이 상태다.

주요 목적은 다음과 같다.

* 현재 문제를 집중해서 읽기
* 4개의 선택지 중 하나 선택
* 필요 시 힌트 확인
* 답안 제출

핵심 기능:

* 현재 문제 번호와 전체 문제 수 표시
* 문제 지문 표시
* 4개 선택지 표시
* 선택지 단일 선택
* 힌트 보기
* 답안 제출
* 북마크
* 이전 화면으로 이동

정책:

* 선택지는 하나만 선택할 수 있다.
* 선택지가 선택되기 전까지 `제출`은 비활성 상태다.
* 사용자가 선택지를 선택하면 `제출`이 활성화된다.
* 사용자가 제출하기 전까지 선택지를 변경할 수 있다.
* 사용자가 제출한 이후에는 해당 Attempt는 변경할 수 없다.

UX 기준:

* 사용자는 문제 지문과 선택지를 한 화면에서 자연스럽게 읽을 수 있어야 한다.
* 선택 여부가 명확히 구분되어야 한다.
* 제출 버튼은 실수 방지를 위해 선택 전 비활성화한다.

---

### 5.3 힌트 보기 화면

문제 풀이 중 사용자가 힌트를 열었을 때의 상태다.

주요 목적은 다음과 같다.

* 정답을 직접 알려주지 않고 사고 방향을 제시
* 문제 요구사항을 정리하도록 유도
* 부적합한 선택지를 제거할 수 있도록 도움

핵심 기능:

* 힌트 카드 확장
* 힌트 닫기
* 힌트 확인 후 답안 선택 및 제출

힌트 내용 예시:

* 핵심 요구사항을 정리해 보세요.
* 고속 업로드, 단일 버킷, 최소한의 운영 복잡성을 고려하세요.
* 수동 운반 또는 전용선 구성처럼 운영 부담이 큰 방식이 적합한지 판단해 보세요.
* AWS에서 제공하는 네트워크 가속 및 병렬 업로드 기능을 떠올려 보세요.

정책:

* 힌트 확인 여부는 학습 이벤트로 기록할 수 있다.
* 힌트 사용이 정답 여부를 제한하지는 않는다.
* 힌트는 정답 선택지를 직접 언급하지 않는다.

UX 기준:

* 힌트는 문제 풀이 흐름을 방해하지 않고 같은 화면 안에서 확장되어야 한다.
* 힌트를 열어도 문제와 선택지의 맥락이 유지되어야 한다.
* 사용자는 힌트를 닫고 다시 문제 풀이에 집중할 수 있어야 한다.

---

### 5.4 정답 및 해설 화면

사용자가 정답을 제출했을 때 정답인 경우의 결과 화면이다.

주요 목적은 다음과 같다.

* 정답 상태를 즉시 확인
* 선택한 답이 왜 맞는지 이해
* 선택지별 해설을 통해 관련 개념 정리
* 다음 문제로 자연스럽게 이동

핵심 기능:

* 정답 상태 표시
* 정답 선택지 표시
* 성공 피드백 제공
* 선택지별 해설 제공
* 다음 문제 이동

정책:

* 정답 Attempt는 풀이 기록에 저장된다.
* 정답 문제는 맞힌 문제 수에 반영된다.
* 사용자가 북마크한 문제라면 북마크 상태는 유지된다.
* 다음 문제 CTA를 통해 다음 미풀이 문제 또는 설정된 학습 큐의 다음 문제로 이동한다.

UX 기준:

* 정답 여부는 직관적으로 인식되어야 한다.
* 해설은 정답만 설명하지 않고 오답 선택지가 왜 부적절한지도 설명해야 한다.
* 사용자는 해설을 읽은 뒤 자연스럽게 다음 문제로 이동할 수 있어야 한다.

---

### 5.5 오답 및 해설 화면

사용자가 정답을 제출했을 때 오답인 경우의 결과 화면이다.

주요 목적은 다음과 같다.

* 오답 상태를 명확히 확인
* 실제 정답과 사용자가 선택한 답을 비교
* 왜 틀렸는지 이해
* 오답 노트에 자동 저장되었음을 알려줌
* 다음 문제 또는 오답 노트로 이동

핵심 기능:

* 오답 상태 표시
* 실제 정답 표시
* 사용자가 선택한 오답 표시
* 오답 노트 자동 저장 안내
* 선택지별 해설 제공
* 오답노트 보기
* 다음 문제 이동

정책:

* 오답 Attempt는 풀이 기록에 저장된다.
* 오답 문제는 오답 노트에 자동 등록된다.
* 동일 문제를 여러 번 틀린 경우 오답 누적 횟수를 기록할 수 있다.
* 오답 노트에는 최근 오답 시점, 선택한 오답, 정답, 해설, 재풀이 상태가 포함될 수 있다.

UX 기준:

* 사용자가 고른 오답과 실제 정답이 명확히 대비되어야 한다.
* 오답 안내는 부정적인 실패감보다 복습 가능성에 초점을 맞춘다.
* 오답노트 보기와 다음 문제 이동 중 사용자가 학습 흐름을 선택할 수 있어야 한다.

## 6. 핵심 사용자 Use Case

### UC-01. 사용자가 앱을 열고 문제 풀이를 시작한다

Actor: 학습자

Precondition:

* 앱에 문제 데이터가 존재한다.
* 사용자의 학습 진행 상태가 존재하거나 신규 상태다.

Main Flow:

1. 사용자가 앱을 실행한다.
2. 시작 화면에서 전체 문제 수, 진행률, 맞힌 문제 수, 틀린 문제 수, 남은 문제 수를 확인한다.
3. 사용자가 `문제 풀기`를 선택한다.
4. 시스템은 다음 풀이 대상 문제를 조회한다.
5. 문제 풀이 화면으로 이동한다.

Postcondition:

* 사용자는 문제 풀이 상태에 진입한다.

---

### UC-02. 사용자가 문제를 풀고 정답을 제출한다

Actor: 학습자

Precondition:

* 사용자는 문제 풀이 화면에 있다.
* 문제에는 4개의 선택지가 존재한다.

Main Flow:

1. 사용자가 문제 지문을 읽는다.
2. 사용자가 선택지 중 하나를 선택한다.
3. 시스템은 선택 상태를 표시하고 제출 버튼을 활성화한다.
4. 사용자가 `제출`을 선택한다.
5. 시스템은 선택한 답과 정답을 비교한다.
6. 정답인 경우 정답 및 해설 화면으로 이동한다.
7. 오답인 경우 오답 및 해설 화면으로 이동한다.

Postcondition:

* Attempt가 저장된다.
* 문제 풀이 결과가 학습 통계에 반영된다.

Exception Flow:

* 선택지를 선택하지 않은 경우 제출할 수 없다.
* 이미 제출된 Attempt는 수정할 수 없다.

---

### UC-03. 사용자가 힌트를 확인하고 문제를 푼다

Actor: 학습자

Precondition:

* 사용자는 문제 풀이 화면에 있다.
* 해당 문제에 힌트가 존재한다.

Main Flow:

1. 사용자가 `힌트 보기`를 선택한다.
2. 시스템은 힌트 카드를 확장한다.
3. 사용자는 힌트를 읽고 문제 요구사항을 재해석한다.
4. 사용자가 선택지를 선택한다.
5. 사용자가 답안을 제출한다.
6. 시스템은 결과 화면으로 이동한다.

Postcondition:

* 힌트 조회 이벤트가 기록될 수 있다.
* Attempt가 저장된다.

Policy:

* 힌트 조회는 정답 판정에 영향을 주지 않는다.
* 힌트는 정답을 직접 노출하지 않는다.

---

### UC-04. 사용자가 오답을 복습한다

Actor: 학습자

Precondition:

* 사용자는 하나 이상의 오답 기록을 가지고 있다.

Main Flow:

1. 사용자가 시작 화면 또는 오답 결과 화면에서 `오답 노트`를 선택한다.
2. 시스템은 오답 문제 목록을 보여준다.
3. 사용자가 특정 오답 문제를 선택한다.
4. 시스템은 문제, 사용자가 선택했던 답, 정답, 해설을 보여준다.
5. 사용자는 해당 문제를 다시 풀거나 해설을 복습한다.

Postcondition:

* 오답 재학습 이력이 기록될 수 있다.
* 사용자의 취약 문제 상태가 갱신될 수 있다.

---

### UC-05. 사용자가 문제를 북마크한다

Actor: 학습자

Precondition:

* 사용자는 문제 풀이 화면 또는 해설 화면에 있다.

Main Flow:

1. 사용자가 북마크 아이콘을 선택한다.
2. 시스템은 해당 문제를 북마크 상태로 변경한다.
3. 사용자는 나중에 북마크 문제만 모아서 확인할 수 있다.

Postcondition:

* 문제의 북마크 상태가 저장된다.

Policy:

* 북마크는 정답 여부와 독립적으로 관리된다.
* 북마크 해제도 가능해야 한다.

---

### UC-06. 사용자가 이어 풀기를 한다

Actor: 학습자

Precondition:

* 사용자의 마지막 학습 위치가 저장되어 있다.

Main Flow:

1. 사용자가 시작 화면에서 `학습 이어하기` 또는 `문제 풀기`를 선택한다.
2. 시스템은 마지막으로 풀던 위치 또는 다음 미풀이 문제를 계산한다.
3. 사용자는 해당 문제 풀이 화면으로 이동한다.

Postcondition:

* 학습 흐름이 이전 위치부터 이어진다.

Policy:

* 마지막 학습 위치가 없으면 첫 번째 미풀이 문제로 이동한다.
* 모든 문제를 풀었다면 복습 또는 오답 풀이로 안내할 수 있다.

## 7. 도메인 모델

### 7.1 Question

시험 문제를 나타내는 핵심 도메인 객체다.

속성 예시:

* questionId
* examCode
* category
* difficulty
* questionText
* choices
* correctChoiceId
* explanation
* hint
* createdAt
* updatedAt

주요 규칙:

* Question은 최소 2개 이상의 Choice를 가져야 한다.
* SAA-C03 객관식 문제는 기본적으로 4개의 Choice를 가진다.
* correctChoiceId는 choices 중 하나여야 한다.
* questionText는 비어 있을 수 없다.
* explanation은 결과 화면에서 제공되므로 필수로 관리하는 것이 바람직하다.

---

### 7.2 Choice

문제의 선택지를 나타낸다.

속성 예시:

* choiceId
* label
* content
* explanation

주요 규칙:

* label은 A, B, C, D 중 하나다.
* 한 Question 안에서 label은 중복될 수 없다.
* Choice explanation은 해설 화면에서 선택지별 판단 근거로 사용된다.

---

### 7.3 Hint

문제 풀이 전 사고를 유도하는 도움말이다.

속성 예시:

* hintId
* questionId
* hints

주요 규칙:

* 힌트는 정답 label을 직접 노출하지 않아야 한다.
* 힌트는 문제 요구사항, 제약 조건, 서비스 특성을 중심으로 구성한다.

---

### 7.4 PracticeSession

사용자의 문제 풀이 세션을 나타낸다.

속성 예시:

* sessionId
* userId
* mode
* currentQuestionId
* startedAt
* lastAccessedAt
* completedAt

mode 예시:

* ALL_QUESTIONS
* INCORRECT_ONLY
* BOOKMARKED_ONLY
* CATEGORY_BASED
* RANDOM

주요 규칙:

* 하나의 세션은 하나의 풀이 모드를 가진다.
* 세션은 현재 풀이 위치를 저장할 수 있어야 한다.
* 사용자는 세션을 통해 이어 풀기를 할 수 있다.

---

### 7.5 Attempt

사용자가 특정 문제에 제출한 답안 기록이다.

속성 예시:

* attemptId
* userId
* questionId
* selectedChoiceId
* correctChoiceId
* result
* usedHint
* submittedAt

result 예시:

* CORRECT
* INCORRECT

주요 규칙:

* Attempt는 제출 이후 변경할 수 없다.
* selectedChoiceId는 해당 Question의 Choice 중 하나여야 한다.
* correctChoiceId는 제출 시점의 정답 기준으로 저장한다.
* 오답인 경우 IncorrectNote 생성 또는 갱신의 근거가 된다.

---

### 7.6 IncorrectNote

사용자의 오답 복습을 위한 도메인 객체다.

속성 예시:

* incorrectNoteId
* userId
* questionId
* latestSelectedChoiceId
* correctChoiceId
* wrongCount
* lastWrongAt
* reviewStatus
* resolvedAt

reviewStatus 예시:

* NEEDS_REVIEW
* REVIEWED
* RESOLVED

주요 규칙:

* 오답 Attempt가 발생하면 자동 생성 또는 갱신된다.
* 같은 문제를 여러 번 틀리면 wrongCount가 증가한다.
* 사용자가 재풀이에서 정답을 맞히면 REVIEWED 또는 RESOLVED로 전환할 수 있다.

---

### 7.7 Bookmark

사용자가 나중에 다시 보고 싶은 문제를 표시한 기록이다.

속성 예시:

* bookmarkId
* userId
* questionId
* createdAt

주요 규칙:

* 동일 사용자는 동일 문제를 중복 북마크할 수 없다.
* 정답 여부와 독립적으로 관리된다.

---

### 7.8 LearningProgress

사용자의 전체 학습 상태를 나타내는 집계 모델이다.

속성 예시:

* userId
* totalQuestionCount
* solvedQuestionCount
* correctCount
* incorrectCount
* remainingCount
* accuracyRate
* lastQuestionId
* updatedAt

주요 규칙:

* solvedQuestionCount는 최소 0 이상이어야 한다.
* correctCount + incorrectCount는 Attempt 정책에 따라 solvedQuestionCount와 일치하거나, 최신 풀이 기준으로 계산될 수 있다.
* remainingCount는 totalQuestionCount - solvedQuestionCount로 계산한다.
* accuracyRate는 solvedQuestionCount가 0일 때 별도 처리한다.

## 8. 주요 도메인 정책

### 8.1 답안 제출 정책

* 답안은 선택지가 선택된 경우에만 제출할 수 있다.
* 하나의 제출은 하나의 Attempt를 생성한다.
* 제출 이후 결과는 변경하지 않는다.
* 정답 판정은 제출 시점의 Question 정답 기준으로 수행한다.

### 8.2 오답 노트 정책

* 사용자가 오답을 제출하면 오답 노트에 자동 저장한다.
* 이미 오답 노트에 존재하는 문제라면 새로 만들지 않고 갱신한다.
* 갱신 시 마지막 선택 오답, 마지막 오답 시각, 누적 오답 횟수를 반영한다.
* 사용자가 이후 재풀이에서 정답을 맞힌 경우에도 오답 기록은 삭제하지 않고 상태를 변경한다.

### 8.3 힌트 정책

* 힌트 조회는 사용자의 풀이 흐름을 보조하는 기능이다.
* 힌트를 조회해도 Attempt 결과에는 불이익을 주지 않는다.
* 단, 분석 목적을 위해 usedHint 값을 Attempt에 저장할 수 있다.
* 힌트는 정답을 직접 알려주지 않는다.

### 8.4 진행률 정책

* 진행률은 전체 문제 대비 풀이 완료 문제 수를 기준으로 계산한다.
* 반복 풀이를 허용하는 경우, 진행률은 “최초 풀이 완료 여부” 기준으로 계산한다.
* 정답률은 제출된 Attempt 기준 또는 문제별 최신 Attempt 기준 중 하나를 선택해야 한다.

권장 방식:

* 홈 화면의 진행률은 문제별 최초 풀이 완료 기준으로 계산한다.
* 통계 화면의 정답률은 최신 Attempt 기준과 누적 Attempt 기준을 분리해서 제공할 수 있다.

## 10. 핵심 상태 전이

### 10.1 문제 풀이 상태

상태:

* NOT_SELECTED
* SELECTED
* HINT_OPENED
* SUBMITTED_CORRECT
* SUBMITTED_INCORRECT

전이:

1. NOT_SELECTED → SELECTED
   사용자가 선택지를 선택한다.

2. NOT_SELECTED → HINT_OPENED
   사용자가 힌트 보기를 선택한다.

3. HINT_OPENED → SELECTED
   사용자가 선택지를 선택한다.

4. SELECTED → SUBMITTED_CORRECT
   사용자가 답안을 제출했고 정답이다.

5. SELECTED → SUBMITTED_INCORRECT
   사용자가 답안을 제출했고 오답이다.

6. HINT_OPENED → NOT_SELECTED
   사용자가 힌트를 닫고 아직 선택하지 않은 상태로 돌아간다.

정책:

* SUBMITTED_CORRECT 또는 SUBMITTED_INCORRECT 상태에서는 선택지를 변경할 수 없다.
* 결과 상태에서는 해설 확인과 다음 액션만 가능하다.

## 13. UX 의사결정 및 트레이드오프

### 13.1 힌트 제공 방식

선택지 A: 힌트를 별도 화면으로 제공
장점은 화면이 깔끔하고 힌트에 집중할 수 있다는 점이다. 단점은 문제와 선택지 맥락이 끊길 수 있다.

선택지 B: 현재 문제 화면 안에서 힌트 카드를 확장
장점은 문제와 선택지 맥락을 유지하면서 힌트를 볼 수 있다는 점이다. 단점은 작은 모바일 화면에서 스크롤 부담이 생길 수 있다.

권장안: 선택지 B
문제풀이 앱에서는 문제 맥락 유지가 중요하므로, 힌트는 현재 화면 안에서 확장하는 방식이 적합하다.

---

### 13.2 오답 저장 방식

선택지 A: 사용자가 직접 오답 노트에 저장
장점은 사용자가 원하는 문제만 저장할 수 있다는 점이다. 단점은 저장을 누락할 가능성이 높고 학습 흐름이 끊긴다.

선택지 B: 오답 자동 저장
장점은 실수를 누락 없이 기록할 수 있고 복습 루프가 자연스럽다. 단점은 오답 노트가 많아질 수 있다.

권장안: 선택지 B
오답 복습은 이 앱의 핵심 학습 루프이므로 자동 저장이 적합하다. 대신 오답 노트에서 필터링, 해결 상태, 숨김 처리를 제공하는 방식으로 관리 부담을 줄인다.

---

### 13.3 정답률 계산 방식

선택지 A: 모든 Attempt 누적 기준
장점은 사용자의 전체 풀이 이력을 정확히 반영한다. 단점은 같은 문제를 여러 번 풀면 통계 해석이 어려워질 수 있다.

선택지 B: 문제별 최신 Attempt 기준
장점은 현재 실력 상태를 더 잘 보여준다. 단점은 과거 실수의 누적 학습량을 드러내기 어렵다.

권장안: 홈 화면은 문제별 최신 Attempt 기준, 통계 화면은 누적 기준과 최신 기준을 분리 제공한다.

## 14. 비기능 요구사항

### 14.1 성능

* 문제 풀이 화면 진입은 빠르게 이루어져야 한다.
* 문제, 선택지, 해설은 화면 전환 시 지연 없이 표시되어야 한다.
* 로컬 캐싱 또는 사전 로딩을 고려할 수 있다.

### 14.2 신뢰성

* 답안 제출 결과는 유실되면 안 된다.
* 네트워크 오류가 발생해도 사용자의 선택과 제출 상태는 보존되어야 한다.
* 오답 자동 저장은 중복 저장보다 멱등 갱신 방식이 적합하다.

### 14.3 접근성

* 주요 CTA는 충분한 터치 영역을 가져야 한다.
* 정답과 오답은 색상만으로 구분하지 않고 텍스트, 아이콘, 배지를 함께 사용해야 한다.
* 긴 문제 지문도 읽기 쉽게 줄 간격과 영역 분리를 유지해야 한다.

### 14.4 확장성

* 향후 다른 AWS 시험 코드로 확장할 수 있어야 한다.
* 문제 카테고리, 난이도, 태그 기반 학습을 추가할 수 있어야 한다.

## 17. Acceptance Criteria

### 17.1 문제 풀이

* 사용자는 문제 지문과 4개의 선택지를 확인할 수 있다.
* 선택지를 선택하기 전에는 제출할 수 없다.
* 선택지를 선택하면 제출 버튼이 활성화된다.
* 제출하면 정답 여부에 따라 올바른 결과 화면으로 이동한다.

### 17.2 힌트

* 사용자는 문제 풀이 중 힌트를 열 수 있다.
* 힌트를 열어도 선택지와 문제 맥락이 유지된다.
* 힌트는 정답을 직접 노출하지 않는다.
* 사용자는 힌트를 닫을 수 있다.

### 17.3 정답 결과

* 정답 제출 시 Correct 상태가 표시된다.
* 실제 정답이 명확히 표시된다.
* 선택지별 해설이 제공된다.
* 사용자는 다음 문제로 이동할 수 있다.

### 17.4 오답 결과

* 오답 제출 시 Incorrect 상태가 표시된다.
* 실제 정답과 사용자가 선택한 오답이 모두 표시된다.
* 오답은 자동으로 오답 노트에 저장된다.
* 사용자는 오답 노트로 이동하거나 다음 문제로 이동할 수 있다.

### 17.5 시작 화면

* 사용자는 전체 문제 수, 진행률, 맞힌 문제 수, 틀린 문제 수, 남은 문제 수를 확인할 수 있다.
* 사용자는 문제 풀이를 시작할 수 있다.
* 사용자는 오답 노트로 이동할 수 있다.
* 마지막 학습 위치가 있는 경우 이어 학습이 가능하다.

## 18. 향후 확장 방향

### 18.1 실전 모의고사 모드

시간 제한, 문제 수 제한, 제출 후 전체 채점 방식의 모의고사 모드를 추가할 수 있다.

### 18.2 개인화 추천

사용자의 오답 패턴, 힌트 사용 여부, 카테고리별 정답률을 기반으로 다음 학습 문제를 추천할 수 있다.

### 18.3 카테고리별 약점 분석

AWS 서비스 영역별로 정답률과 오답률을 분석하여 사용자가 취약한 영역을 확인할 수 있도록 한다.

### 18.4 해설 품질 고도화

정답 해설뿐 아니라 왜 다른 선택지가 부적절한지 더 구체적으로 제공하고, 관련 AWS 개념 요약을 연결할 수 있다.

### 18.5 오프라인 학습

문제와 해설 데이터를 로컬에 저장하여 네트워크 없이도 학습할 수 있도록 확장할 수 있다.
